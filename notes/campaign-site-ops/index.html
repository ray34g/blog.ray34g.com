短期キャンペーンサイトを静的サイトとして運用する設計メモ

公開日が決まっている多言語キャンペーンサイトを、静的サイト生成と編集ワークフローで安全に運用するための考え方。

このノートは、実運用されたキャンペーンサイトの経験を匿名化して整理したものです。固有名詞、関係団体名、内部 URL、アカウント名、実際の運用手順のうち公開に向かない情報は意図的に省いています。

前提

対象は、公開日と終了時期がある短期キャンペーンサイトです。要件はシンプルに見えても、実際には次のような制約が重なります。

  • 日英など複数言語で同じ情報を更新する
  • 非エンジニアがニュースや固定ページを編集する
  • 公開前に関係者がレビューできる
  • 本番公開後も短期間で修正が入る
  • 画像、フォント、動画などの表示品質を保つ
  • 外部サーバーや Pages artifact など、複数の配信先を扱う

この条件では、サイトそのものの実装だけでなく、更新、確認、公開、障害対応までを一つの運用基盤として考える必要があります。

静的サイトに寄せる理由

キャンペーンサイトは、常時ログインが必要なアプリケーションではありません。表示の安定性、配信コスト、復旧しやすさを優先すると、静的サイト生成はかなり相性が良い選択でした。

一方で、静的サイトに寄せるだけでは編集体験が弱くなります。そこで、コンテンツ編集は CMS に寄せ、公開物はビルド済み HTML とアセットに落とす構成にします。

Editor
  -> CMS
  -> Git repository
  -> CI build
  -> Preview / Release / Production

この流れにすると、変更履歴、レビュー、差し戻し、再ビルドが Git の上に乗ります。小さなチームでも、公開事故を減らすための足場を作りやすくなります。

設計で重視したこと

まず、編集者が触る情報と、開発者が保守する情報を分けます。ニュース本文、FAQ、会場情報、寄付先情報のようなものはコンテンツとして扱い、テンプレート、ビルド、検証、配信設定は開発側の責務にします。

次に、公開前確認を必ず通れる構成にします。CMS から保存された変更が即本番に出る構成は速い反面、短期キャンペーンでは関係者レビューとの相性がよくありません。preview と production を分け、差分を確認してから公開できるようにします。

最後に、運用ドキュメントを成果物として扱います。セットアップ、アカウント管理、編集手順、公開手順、障害時の連絡先を文書化しておくと、サイト公開後の問い合わせが減り、引き継ぎもしやすくなります。

ポートフォリオとしての見せ方

実名を出せない場合でも、価値は落ちません。むしろ、制約を抽象化して説明できること自体が設計力の証明になります。

公開できる範囲では、次のような粒度で記録するとよさそうです。

  • どんな制約があったか
  • なぜ静的サイト生成を選んだか
  • 編集者と開発者の責務をどう分けたか
  • preview、release、production をどう考えたか
  • 品質確認をどこまで自動化したか
  • 次に再利用するなら何を抽出するか

実名の代わりに、判断の再現性を置く。これが匿名ケーススタディの中心になります。
